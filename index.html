Glossary of Key Terms

Organized to match the job description, section by section — query skills first, then data validation, then the wealth-management domain knowledge, then the tooling and governance around it.

SQL & Query Design
SQL (Structured Query Language)Language

The standard language for querying and manipulating data stored in relational databases. "Expert-level SQL" means being fluent well beyond basic SELECT statements — comfortable combining multiple tables, writing readable multi-step logic, and reasoning about performance on large datasets.

In practiceSQL is the primary tool for both building analysis and validating that a pipeline's output actually matches what it should.
Multi-Table JoinsSQL Skill

Combining rows from two or more tables based on a related column — for example, joining an accounts table to a transactions table on account ID. An INNER JOIN keeps only matching rows; a LEFT JOIN keeps every row from the first table even without a match, which matters when you specifically need to see what's missing.

In practiceWealth-management questions almost always span several tables — accounts, holdings, transactions, fees — so comfortable multi-table joins are a baseline expectation, not a bonus skill.
CTE (Common Table Expression)SQL Skill

A named, temporary result set defined with a WITH clause at the start of a query, which the rest of the query can then reference like a table. CTEs break a complex query into readable, logical steps instead of one dense nested subquery, and can be chained one after another.

In practiceCTEs are how you make a multi-step reconciliation or aggregation query something a reviewer can actually follow.
Window FunctionsSQL Skill

Functions that calculate a value across a set of related rows — a "window" — without collapsing those rows into a single output row the way GROUP BY does. Common examples include ROW_NUMBER(), running totals, and calculating period-over-period change using LAG()/LEAD().

In practiceWindow functions are exactly what's needed to calculate something like month-over-month portfolio performance without losing the individual transaction-level detail.
SubqueriesSQL Skill

A query nested inside another query, used to filter, compare against, or supply values to the outer query. Subqueries can appear in the WHERE, FROM, or SELECT clause, and are often an alternative to — or combined with — a CTE.

In practiceKnowing when a subquery is clearer than a CTE (and vice versa) is part of what "analytical query design" is actually testing for.
Analytical Query DesignSkill

The broader skill of structuring a query — or a sequence of queries — to answer a real business question efficiently and readably, not just technically correctly. This includes choosing the right join strategy, breaking logic into clear steps, and writing something a teammate could pick up and understand later.

In practiceThis is the difference between a query that happens to return the right numbers and one that's maintainable, explainable, and fast on a large dataset.
Requirements, Discovery & Validation
Functional Requirements & Acceptance CriteriaProcess

Functional requirements describe what a piece of work needs to do, gathered from business stakeholders in their own language. Acceptance criteria translate that into specific, testable conditions — the concrete checklist used to determine whether the finished work actually satisfies the requirement.

In practiceA data analyst's job is often turning a stakeholder's plain-language ask ("show me fee revenue by advisor") into acceptance criteria precise enough for someone else to build and test against.
Data Discovery & ProfilingProcess

The process of exploring an unfamiliar dataset before relying on it — understanding what tables and fields exist, how they relate, what a "normal" value looks like, and where the gaps, duplicates, or inconsistencies are. Profiling typically means running summary statistics, checking for nulls, and spot-checking against known-good records.

In practiceThis is usually the first real step on any new dataset, well before writing analysis — you can't trust a query's output if you don't understand the data feeding it.
ETL (Extract, Transform, Load)Process

The general pattern for moving data from source systems into a destination system usable for analysis: Extract pulls raw data out, Transform cleans and reshapes it according to business rules, and Load writes the result into its destination (a table, a file, another system). "ELT" is a common variant where the raw data is loaded first and transformed afterward, inside the destination platform.

In practiceValidating an ETL process means checking each of those three steps independently — did extraction miss anything, did the transform logic apply correctly, did the load land intact.
Reconciliation (Source vs. Target)Validation

Comparing data at its origin (the source) against the same data after it has moved through a pipeline (the target) to confirm nothing was lost, duplicated, or altered incorrectly along the way. This might mean comparing row counts, sums of key fields (like total AUM), or matching individual records one-to-one.

In practiceReconciliation is the core technique for proving a pipeline's business logic is correct — not just that it ran without errors, but that its output actually ties back to the source.
Test Cases (Data Validation)Validation

Specific, documented scenarios written to verify a pipeline or piece of logic behaves correctly — including edge cases, not just the typical case. A good test case states the input, the expected output, and how to check it, so someone else can run it and get the same result.

In practiceWriting test cases for a new custodian feed or business-logic rule is how you catch problems before they reach a client-facing report.
Data Quality IssuesConcept

Problems in a dataset that make it inaccurate, incomplete, or inconsistent — missing values, duplicate records, mismatched formats between systems, outdated reference data, or values that fall outside a plausible range. Documenting these clearly (what's wrong, how big the impact is, which records are affected) is as important as finding them.

In practiceClear documentation of a quality issue is what lets a business stakeholder or engineer actually act on it, rather than just being told "something's off."
Wealth Management Data Concepts
AUM (Assets Under Management)Metric

The total market value of the investments a firm or advisor manages on behalf of clients, typically calculated by summing the current value of all client holdings. AUM is a central metric in wealth management — it usually drives both how a firm's success is measured and how many fee structures are calculated.

In practiceReconciling AUM between a custodian feed and an internal system is one of the most common validation exercises in this kind of role.
Portfolio HoldingsConcept

The specific set of investments — stocks, bonds, funds, cash, alternatives — held in an account at a given point in time, including quantity and market value for each position. Holdings data is typically snapshot-based (as-of a date) rather than continuous.

In practiceHoldings are usually the largest and most detail-heavy dataset in a wealth-management pipeline, since every account can hold dozens of individual positions.
Performance ReturnsConcept

How a portfolio's value has changed over a period, usually expressed as a percentage and often benchmarked against a comparison index. Calculating returns correctly requires accounting for cash flows in and out of the account (deposits, withdrawals) so the number reflects investment performance, not just account activity.

In practicePerformance calculations are a common source of reconciliation disputes, since small differences in methodology (time-weighted vs. money-weighted return) can produce genuinely different, both "correct," numbers.
FeesConcept

Charges applied to a client's account — most commonly an advisory or management fee calculated as a percentage of AUM, billed on a regular schedule (often quarterly). Fee data needs to tie precisely back to the AUM and billing methodology it was calculated from, since it directly affects client statements and revenue reporting.

In practiceFee logic is a classic candidate for a validation test case — a small error in the calculation compounds across every client account it touches.
TransactionsConcept

Individual events that change an account's holdings or cash balance — buys, sells, dividends, deposits, withdrawals, fee debits, and transfers. Transaction-level data is what holdings and performance are ultimately built from, so errors here tend to cascade downstream.

In practiceWhen a holdings or performance number looks wrong, tracing it back to the underlying transactions is usually the first debugging step.
Custodians & Wealth-Tech Platforms
Custodian FeedsData Source

A custodian is a financial institution that holds and safeguards client assets and handles trade settlement and recordkeeping. Custodians deliver data — holdings, transactions, balances — to advisory firms as structured feed files, typically on a daily basis, and each custodian tends to have its own file format and quirks.

The custody market is concentrated: Charles Schwab is generally the largest custodian for independent advisory firms, followed by Fidelity, then BNY Pershing and LPL Financial — together these four control the large majority of custodied assets in the U.S. Reconciling a custodian's feed against internal records is a routine, recurring task, since format changes or dropped records on the custodian side happen and need to be caught.

In practiceExposure to more than one custodian's feed format (Schwab, Pershing, Fidelity) is exactly what lets an analyst spot when something is off before it reaches a client report.
Wealth & Portfolio Management PlatformsVendor

Software platforms that aggregate holdings across custodians and asset types to produce performance, allocation, and client-facing reports. Addepar is generally considered the benchmark for complex, multi-entity, alternative-asset portfolios at the high-net-worth level. Orion Advisor Tech is a broader all-in-one platform that also bundles CRM and trading. SS&C Black Diamond is known for polished, branded client reporting with a large integrations network and a lighter implementation footprint than Addepar.

In practiceWhichever platform is in use, an analyst's work usually ends with data landing in one of these — so understanding the shape of data they expect matters directly.
Python & Data Tooling
Python for Data AnalysisLanguage

Python is a general-purpose programming language widely used for ad hoc data exploration, scripting, and automation because of its readable syntax and mature data-analysis libraries. In this kind of role it typically supplements SQL — used for tasks SQL is clumsier at, like complex reshaping, statistical checks, or generating a one-off report.

In practiceA common workflow is pulling data with SQL, then using Python for the last mile of analysis or validation logic that's awkward to express in a query.
pandasLibrary

The standard Python library for working with tabular data — think of it as a spreadsheet inside Python. Its core object, the DataFrame, supports filtering, joining, grouping, and reshaping data with code instead of clicking through a UI, and it's the most common tool for ad hoc data profiling and validation scripts.

In practicepandas is typically where a reconciliation script actually lives — pulling both source and target data, then comparing them programmatically instead of by hand.
NumPyLibrary

A foundational Python library for fast numerical computation, built around efficient array operations. pandas itself is built on top of NumPy — so even when an analyst doesn't call NumPy directly, it's usually doing the underlying math whenever pandas performs a calculation across a column of numbers.

In practiceNumPy shows up directly when doing vectorized calculations — like computing returns across thousands of accounts at once — faster than looping row by row.
PySparkLibrary

The Python interface to Apache Spark, the distributed data-processing engine. PySpark lets an analyst write familiar-feeling Python/pandas-style code that actually runs across a cluster of machines in parallel, which matters once a dataset is too large for a single machine's memory to handle efficiently.

In practicePySpark is the tool of choice once profiling or transformation work outgrows what pandas can comfortably handle on one machine — commonly inside Databricks.
Databricks & Cloud Data Platforms
Data Platform (Lakehouse)Platform

A unified data platform — often called a "lakehouse" — that combines large-scale data storage with the compute needed to process, transform, and analyze that data in one place, instead of storage and processing living in separate, disconnected tools. Databricks is a leading example, built on the open-source Apache Spark project and the Delta Lake storage format.

In practiceThis is typically where a wealth-management analyst would query and profile large datasets — custodian feeds, transaction history — using SQL or PySpark.
Apache SparkEngine

An open-source distributed data-processing engine that splits large computations across many machines so they run in parallel, instead of on one machine at a time. Spark can process data in-memory, which makes it fast for large-scale transformations, and it supports multiple languages — Python, SQL, Scala, and R — often in the same notebook.

In practiceDatabricks was built by Spark's original creators, and Spark is the engine actually executing the query or transformation underneath every Databricks job.
Delta LakeStorage Format

An open-source storage layer that adds database-like reliability to files sitting in cloud storage: ACID transactions (a write either fully succeeds or fully fails), schema enforcement and evolution, and "time travel" — the ability to query a table as it existed at an earlier point, invaluable for debugging a bad run.

In practiceDelta Lake is the default table format on Databricks — the pipeline's tables are typically Delta tables rather than plain files.
Unity CatalogGovernance

Databricks' centralized governance layer for data assets: fine-grained access control (down to specific columns), a searchable catalog of what data exists and where, lineage tracking showing which upstream tables fed a given result, and audit logging of who accessed what.

In practiceThis is what lets a team prove exactly where a client-facing number came from — a direct extension of the data governance the role requires.
Medallion Architecture (Bronze / Silver / Gold)Pattern

A data design pattern that organizes a lakehouse into progressively refined layers. Bronze holds raw, largely untouched data exactly as it arrived from source systems. Silver cleans, deduplicates, and standardizes that data into a validated view. Gold holds business-ready, aggregated tables shaped for reporting and downstream use.

In practiceReconciliation is often done layer by layer — checking that Silver still matches Bronze, then that Gold still matches Silver, to isolate exactly where a discrepancy was introduced.
Batch vs. StreamingConcept

Two ways to move data through a pipeline. Batch processing handles data in scheduled, discrete chunks — for example, a job that runs once every night on the prior day's custodian feed. Streaming processes data continuously as it arrives, used when near-real-time freshness genuinely matters.

In practiceMost wealth-management reporting is well served by nightly batch processing; daily custodian feeds are themselves a batch pattern.
Cluster & NotebookCompute

A cluster is the actual compute resources — virtual machines — running the processing work, scaling up or down with the workload. A notebook is the interactive, cell-by-cell coding environment used to write and run that code, mixing Python, SQL, and documentation in one place.

In practiceAd hoc data discovery and profiling typically happens directly in a notebook against a small cluster before any logic gets scheduled to run automatically.
Cloud Data Platforms (AWS, Azure, Snowflake)Platform

Beyond Databricks, the other major cloud platforms an analyst may encounter: AWS and Azure are the underlying cloud infrastructure providers Databricks and other tools often run on top of, while Snowflake is a SQL-first, fully-managed data warehouse — strong for analysts running dashboards and reports, with less emphasis on large-scale engineering than a Spark-based platform.

In practiceMany organizations run more than one of these together — for example, Databricks for ingestion and transformation, with Snowflake or a BI tool serving the final reporting layer.
Governance, Compliance & Risk
Data GovernanceConcept

The set of policies, roles, and controls that determine who can access data, how it's classified, how its quality is maintained, and how changes to it are tracked. Governance is what turns "we have a lot of data" into "we can trust and control this data."

In practiceIn financial services this isn't optional — governance decisions directly support regulatory requirements around data accuracy and access control.
Data Quality FrameworkConcept

A structured, repeatable approach to measuring and maintaining data quality — typically defined across dimensions like accuracy, completeness, consistency, timeliness, and uniqueness, with specific rules and thresholds for each. A framework turns ad hoc "this looks wrong" checks into a consistent, monitored process.

In practiceRather than reinventing checks for every new dataset, a mature team applies the same quality dimensions consistently across all of its pipelines.
Regulatory Compliance (Financial Services)Concept

The requirement that data handling, recordkeeping, and reporting meet standards set by financial regulators — in the U.S., bodies like the SEC and FINRA. This includes rules around recordkeeping retention, accurate client reporting, and data security, all of which shape how a pipeline is designed, not just how it's used.

In practiceA data quality issue in a regulated firm isn't just an internal inconvenience — it can have compliance implications, which is part of why documentation and reconciliation matter so much.
Predictive Analytics & ML ApplicationsConcept

Using historical data to build models that estimate future outcomes or classify clients, rather than simply reporting what already happened. Two common financial-services examples: client segmentation, which groups clients by behavior or characteristics to tailor service or outreach, and risk modeling, which estimates the likelihood or size of a potential loss or adverse event.

In practiceThis is listed as a "plus," not a core requirement — a data analyst is more likely to prepare clean, reliable data that a data scientist then models than to build the model itself, but basic familiarity helps in cross-functional conversations.
Certifications
CFA (Chartered Financial Analyst)Certification

A globally recognized credential from the CFA Institute focused on investment analysis, portfolio management, and ethics, earned through three graduate-level exams plus qualifying work experience. It's considered the standard credential for investment management roles broadly, not specific to data or performance-reporting work.

In practiceA CFA signals broad investment knowledge — useful context for a data analyst, though the role itself is centered on data and pipeline skills rather than investment decision-making.
CIPM (Certificate in Investment Performance Measurement)Certification

Also offered by the CFA Institute, but narrower and more directly relevant to this kind of role: CIPM focuses specifically on investment performance measurement, attribution, and the GIPS (Global Investment Performance Standards) that govern how performance is calculated and presented. It's earned through two exam levels and has no work-experience prerequisite.

In practiceOf the two, CIPM maps more directly onto the performance-returns and reconciliation work this role actually does day to day.
Engineering Practices & Tooling
CI/CD PipelinePractice

"CI" (Continuous Integration) means code changes are automatically tested and merged frequently, rather than sitting unintegrated for a long time. "CD" (Continuous Delivery/Deployment) means those changes are automatically packaged and pushed toward production once they pass. Together, a CI/CD pipeline automates the path from a code change to a running, tested change — reducing manual, error-prone release steps.

In practiceMoving data pipeline code onto CI/CD is what makes it safe to change business logic without someone manually deploying it and hoping nothing breaks.
SFTP (Secure File Transfer Protocol)Delivery Method

A network protocol for securely transferring files between systems, encrypting both the login credentials and the file contents in transit. It's a common, standardized way for one organization to deliver files to another on a schedule — for example, dropping a daily reconciliation file for a vendor to pick up.

In practiceMoving file delivery to SFTP is typically about standardizing and securing a handoff that may have previously relied on a less secure or more manual method.
Infrastructure as Code (Terraform)Practice

The practice of defining infrastructure — servers, networking, permissions, cloud resources — in version-controlled configuration files instead of setting it up by hand through a UI. Terraform is a widely used tool for this: you describe the infrastructure you want, and it creates, updates, or tears it down to match, consistently and repeatably.

In practiceWhether a team uses Terraform (or a similar tool) versus configuring environments by hand is a real, meaningful choice — it determines how easy it is to reliably reproduce a dev/test/prod environment.
Azure DevOpsPlatform

A Microsoft platform that bundles several software delivery tools together: source control (repos), CI/CD pipelines, work-item/board tracking, and test management, all in one product. It serves a similar overall purpose to combining separate tools like GitHub, Jenkins, and Jira — just packaged as one integrated suite.

In practiceWhich platform a team standardizes on shapes daily workflow — code review, deployment approvals, and task tracking all happen inside it.
Source Control & Issue Tracking (Bitbucket, GitHub, Jira)Tooling

Bitbucket and GitHub are both Git-based source control platforms for storing and collaborating on code, with pull requests, code review, and integrated CI/CD; Bitbucket is more common in shops already standardized on Atlassian's tools. Jira, also from Atlassian, is separate from either — it's a work-tracking tool for tickets, sprints, and backlogs, used to plan and track the work itself rather than store the code.

In practiceKnowing which of these a team actually uses (and whether they're separate products or bundled, as in Azure DevOps) is basic but essential onboarding information.
Static Code Analysis (SonarQube)Tooling

Automatically scanning source code for bugs, security vulnerabilities, and style/maintainability issues without actually running the code. SonarQube is a common tool for this, typically wired into a CI/CD pipeline so every code change gets scanned automatically before it can be merged.

In practiceThis is a quality gate that catches issues earlier and more consistently than relying on manual code review alone.
Databricks Workspaces & Non-Prod CatalogsEnvironment

A Databricks workspace is a separate environment (with its own clusters, notebooks, and access controls) — teams commonly maintain separate workspaces for development, testing, and production so that work-in-progress code never touches live data or live jobs. Unity Catalog can likewise be scoped per environment, so a non-production catalog holds non-production data and schemas, distinct from the catalog production jobs actually read from and write to.

Without this separation, engineers end up editing and testing directly against production — a real risk, since an untested change can immediately affect live client-facing data with no safety net.

In practiceStanding up a genuine dev/test/prod split (separate workspaces, and ideally a separate non-prod catalog) is usually a prerequisite for safely adopting CI/CD in the first place.
Householding & Requirements Scoping
HouseholdingWealth Concept

The practice of grouping a client's related accounts — spouses, family members, trusts, multiple account types — into a single "household" view, rather than looking at each account in isolation. Householding supports a more complete picture of a family's total wealth, risk exposure, and tax situation, and increasingly drives coordinated, household-level rebalancing and billing rather than just account-level reporting.

A common operational challenge: a household and an individual client within it can look similar or ambiguous in a system's UI, so accidentally selecting the wrong level (household vs. an individual client inside it) produces a genuinely different number — since a household's total net worth is not the same figure as any one client's net worth within it.

In practiceGetting householding logic right is a recurring source of gaps between what a portfolio-reporting vendor supports natively and what a firm actually needs.
Logic Enhancement vs. Utility (Scoping Framework)Process

A way of triaging a functional gap once it's identified. First check whether the vendor platform already handles it natively — if so, no build is needed. If not, decide whether it belongs in the core pipeline as a logic enhancement (a change to the actual business logic/data pipeline), or whether it's better solved with a purpose-built utility — a separate tool built specifically to handle that gap, kept outside the core pipeline logic.

The framework exists because a utility can technically be built to handle almost anything, which makes it tempting to overuse; the better test is business impact and whether the fix genuinely belongs inside the core data logic or is a narrower, separable problem.

In practiceApplying this triage consistently — rather than defaulting to "build a utility" every time — is what keeps a data pipeline's core logic from becoming a patchwork of one-off exceptions.
Current State vs. Future State AnalysisProcess

A standard business-analysis technique: documenting exactly how something works today (current state) alongside exactly how it should work once a fix or enhancement is complete (future state), so a gap has a clear, agreed starting point and end point rather than an open-ended discussion.

In practiceDefining the future state explicitly — including which parts will be handled by the vendor's interface versus a custom-built utility — is what turns a requirements conversation into something a team can actually estimate and schedule.
